## <a name="browser-definition-files-updated"></a>Обновить файлах определения браузеров

Файлы описания браузеров обновлены: теперь они включают сведения о новых и обновленных браузерах и устройствах. Устаревшие браузеры и устройства (например, Netscape Navigator) удалены; добавлены новые браузеры и устройства (например, Google Chrome и Apple iPhone).

## <a name="systemwebmobiledll-removed-from-root-web-configuration-file"></a>System.Web.Mobile.dll удалены из корневого файла веб-конфигурации

В предыдущих версиях ASP.NET, ссылку на сборку System.Web.Mobile.dll был включен в корне `Web.config` файла в **сборки** раздела. Чтобы улучшить производительность, ссылку на эту сборку удалена.

## <a name="aspnet-request-validation"></a>Проверка запросов ASP.NET

Средство проверки запросов в ASP.NET обеспечивает определенный уровень защиты по умолчанию от атак межсайтовых сценариев (XSS). В предыдущих версиях ASP.NET проверка запроса была включена по умолчанию. Тем не менее, он применяется только к страницам ASP.NET (`.aspx` файлы и их файлы класса) и только при выполнении этих страниц.

В ASP.NET 4, по умолчанию включена проверка запросов для всех запросов, так как он используется перед **BeginRequest** этап HTTP-запроса. В результате проверки запроса касается запросов для всех ресурсов ASP.NET, не только запросы страниц .aspx. Сюда входят запросы, такие как вызовы веб-службы и пользовательские обработчики HTTP. Проверка запроса уже активен при чтении пользовательские модули HTTP содержимое HTTP-запроса.

В результате запроса могут возникать ошибки проверки теперь для запросов, которые ранее не был запущен ошибок. Чтобы вернуться к поведению средство проверки запросов ASP.NET 2.0, добавьте следующий параметр в `Web.config` файла:

## <a name="configuration-errors-related-to-new-aspnet-4-root-configuration"></a>Конфигурация ошибок, связанных с новой конфигурации ASP.NET 4 корня

Корневых файлов конфигурации ( `machine.config` файла и корневой каталог `Web.config` файл) для .NET Framework 4 (и поэтому ASP.NET 4) были обновлены для включения большая часть параметров шаблона, в ASP.NET 3.5 был найден в приложение `Web.config` файлов. Из-за сложности в управляемых системах конфигурации IIS 7 и IIS 7.5 выполнение приложений ASP.NET 3.5 в ASP.NET 4 и в IIS 7 и IIS 7.5 может привести к ASP.NET или IIS ошибки конфигурации.

Рекомендуется обновить приложения ASP.NET 3.5 до ASP.NET 4 с помощью средства обновления проекта в Visual Studio 2010, если это возможно. Visual Studio 2010 автоматически изменяет приложения ASP.NET 3.5 `Web.config` файл, содержащий соответствующие параметры для ASP.NET 4.

Однако это поддерживается для запуска приложений ASP.NET 3.5, с помощью .NET Framework 4 без повторной компиляции. В этом случае может возникнуть необходимость вручную изменить приложения `Web.config` файл перед запуском приложения в .NET Framework 4 и с IIS 7 или IIS 7.5.

В следующих двух разделах описаны изменения, которые может потребоваться внести для различных сочетаний программного обеспечения.

**Windows Vista с пакетом обновления 1 или Windows Server 2008 с пакетом обновления 1, где установлены ни исправление KB958854 ни с пакетом обновления 2.** В этой конфигурации системы конфигурации IIS 7 неверно объединяет конфигурации управляемого приложения, сравнивая уровня приложения `Web.config` файла в ASP.NET 2.0 `machine.config` файлов. Из-за этого, на уровне приложений `Web.config` файлы из .NET Framework 3.5 или более поздней версии должны иметь **system.web.extensions** определение раздел конфигурации (элемент), чтобы не вызвать ошибку проверки IIS 7.

Тем не менее, вручную изменять уровня приложения `Web.config` записи файлов, которые не соответствуют исходной стандартным определениям раздела конфигурации, которые появились в Visual Studio 2008 вызовет ошибки конфигурации ASP.NET. (Записи конфигурации по умолчанию, созданными с помощью Visual Studio 2008 работать правильно.) Общая проблема состоит в том изменены вручную `Web.config` опустить файлы **allowDefinition** и **requirePermission** атрибутов конфигурации, которые находятся на различных раздела конфигурации определения. В результате несоответствия между сокращенным разделом конфигурации на уровне приложения `Web.config` файлы и полное определение в ASP.NET 4 `machine.config` файла. В результате во время выполнения система конфигурации ASP.NET 4 вызывает ошибку конфигурации.

**Windows Vista SP2, Windows Server 2008 SP2, Windows 7, Windows Server 2008 R2, а также Windows Vista с пакетом обновления 1 и Windows Server 2008 SP1, где установлено исправление KB958854.**

В этом случае система Конфигурация собственного IIS 7 и IIS 7.5 возвращает ошибку конфигурации, так как он выполняет сравнение текста на **тип** атрибут, определенный для обработчика раздела конфигурации управляемого. Так как все `Web.config` файлы, созданными с помощью Visual Studio 2008 и Visual Studio 2008 SP1 имеют «3.5» в строку типа для **system.web.extensions** (и связанных) обработчики разделов конфигурации, а потому, что ASP.NET 4 `machine.config` файл имеет «4.0» **тип** атрибут же обработчики разделов конфигурации приложений, которые всегда создаются в Visual Studio 2008 или Visual Studio 2008 SP1 не проходят проверку конфигурации в IIS 7 и IIS 7.5.

<a id="0.1__Toc251910248"></a>

### <a name="resolving-these-issues"></a>Для устранения подобных проблем

В первом случае достаточно выполнить обновление уровня приложения `Web.config` файла, включая стандартного текста конфигурации из `Web.config` файла, автоматически созданные Visual Studio 2008.

Второе решение для первого сценария — для установки с пакетом обновления 2 для Vista или Windows Server 2008 на компьютере или установите исправление KB958854 ([https://support.microsoft.com/kb/958854](https://support.microsoft.com/kb/958854)) чтобы исправить неверные поведение при объединении конфигурации системы конфигурации IIS. Однако после выполнения любого из этих действий, приложения могут возникнуть из-за проблемы, описанные во втором случае ошибка конфигурации.

Во втором случае достаточно выполнить удалите или закомментируйте все **system.web.extensions** определения раздела конфигурации и раздел конфигурации группы определений из уровня приложения `Web.config` файла. Определения этих обычно находятся в верхней части уровня приложения `Web.config` файла и могут идентифицироваться по **configSections** элемента и его дочерних элементов.

В обоих случаях рекомендуется также вручную удалить **system.codedom** статьи, несмотря на то, что это не является обязательным.

Приложения ASP.NET 4, которые настроены как дочерние для приложений, использующих предыдущие версии ASP.NET, могут не запускаться из-за ошибок конфигурации или компиляции. Следующий пример показывает структуру каталогов для уязвимых приложений.

## <a name="aspnet-4-web-sites-fail-to-start-on-computers-where-sharepoint-is-installed"></a>В ASP.NET 4 веб-сайтов не запускаться на компьютерах, где установлен SharePoint

Веб-серверы, на которых выполняется служба SharePoint имеют `Web.config` файла, который развернут в корневом веб-сайта SharePoint (например, `c:\inetpub\wwwroot\web.config` для по умолчанию веб-сайта). В этом `Web.config` файл SharePoint задает пользовательские частичного доверия уровня с именем WSS\_минимальной.

При попытке запустить сайт ASP.NET 4 Web, развернутой как дочерний элемент этого типа веб-сайта SharePoint, появится следующая ошибка:

`Could not find permission set named 'ASP.NET'.`

Эта ошибка возникает, поскольку инфраструктуру разграничения доступа кода ASP.NET 4 ищет набор разрешений с именем ASP.NET. Однако частичного доверия файл конфигурации, на который ссылается WSS\_минимальной не содержит все наборы разрешений с таким именем.

В настоящее время не имеется версия SharePoint, совместимую с ASP.NET. Поэтому не следует пытаться запустить сайт ASP.NET 4 Web как к дочернему сайту под SharePoint Web сайтов.

## <a name="the-httprequestfilepath-property-no-longer-includes-pathinfo-values"></a>Свойство HttpRequest.FilePath больше не содержит значения PathInfo

Предыдущие версии ASP.NET включены **PathInfo** значение в значение, возвращаемое из различных путь связанных свойств файла, включая **HttpRequest.FilePath**,  **HttpRequest.AppRelativeCurrentExecutionFilePath**, и **HttpRequest.CurrentExecutionFilePath**. В ASP.NET 4 больше не содержит **PathInfo** значение в значения, возвращаемые из этих свойств. Вместо этого **PathInfo** информация доступна в **HttpRequest.PathInfo**. Рассмотрим следующий фрагмент URL-адреса:

`/testapp/Action.mvc/SomeAction`

В более ранних версиях ASP.NET **HttpRequest** свойства имеют следующие значения:

**HttpRequest.FilePath**:`/testapp/Action.mvc/SomeAction`

**HttpRequest.PathInfo**: (пустой)

В ASP.NET 4 **HttpRequest** свойства вместо имеют следующие значения:

**HttpRequest.FilePath**:`/testapp/Action.mvc`

**HttpRequest.PathInfo**:`SomeAction`

## <a name="aspnet-20-applications-might-generate-httpexception-errors-that-reference-eurlaxd"></a>ASP.NET 2.0 приложения могут вызывать ошибки HttpException, которые ссылаются на eurl.axd

После включения ASP.NET 4 в IIS 6 приложений ASP.NET 2.0, работающих в IIS 6 (в Windows Server 2003 или Windows Server 2003 R2) могут возникнуть ошибки из следующего:

`System.Web.HttpException: Path '/[yourApplicationRoot]/eurl.axd/[Value]' was not found.`

Эта ошибка возникает, поскольку при ASP.NET обнаруживает, что веб-сайт настроен для использования ASP.NET 4, собственный компонент ASP.NET 4 передает URL-адрес без расширения управляемой части ASP.NET для дальнейшей обработки. Тем не менее если виртуальные каталоги, которые ниже 4 веб-сайта ASP.NET настроены для использования ASP.NET 2.0, обработка URL-адрес без расширений в результатом является способом изменения URL-адрес, содержащий строку «eurl.axd». Этот измененный URL-адрес затем отправляется в приложение ASP.NET 2.0. ASP.NET 2.0 не распознает формат «eurl.axd». Таким образом, ASP.NET 2.0 пытается найти файл с именем `eurl.axd` и выполните его. Так как файл не существует, запрос завершится ошибкой с **HttpException** исключение.

Можно обойти эту проблему, с помощью одного из следующих вариантов.

### <a name="option-1"></a>Вариант 1

Если ASP.NET 4 не требуется для запуска веб-сайта, измените сопоставление сайта для использования ASP.NET 2.0.

### <a name="option-2"></a>Вариант 2.

Если нужно запустить веб-сайт ASP.NET 4, переместите любые дочерние виртуальные каталоги ASP.NET 2.0 на другой веб-сайт, который сопоставлен ASP.NET 2.0.

### <a name="option-3"></a>Вариант 3.

Если нецелесообразно повторно сопоставить веб-сайт ASP.NET 2.0 или изменение расположения виртуального каталога, явно отключите URL-адресов без обработки в ASP.NET 4. Используйте следующую процедуру:

1. В реестре Windows откройте следующий узел:

`HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\ASP.NET\4.0.30319.0`

1. Создайте новый **DWORD** значение с именем **EnableExtensionlessUrls**.
2. Задать **EnableExtensionlessUrls** значение 0. Это приведет к отключению без расширений поведения URL-адрес.
3. Сохраните значение реестра и закройте редактор реестра.
4. Запустите **iisreset** средство командной строки, что приведет к тому IIS для чтения нового значения реестра.

> [!NOTE]
> Установка **EnableExtensionlessUrls** значение 1 включает поведение без расширений URL-адрес. Это параметр по умолчанию, если значение не указано.

## <a name="event-handlers-might-not-be-not-raised-in-a-default-document-in-iis-7-or-iis-75-integrated-mode"></a>Обработчики событий могут не не вызываться в документ по умолчанию в IIS 7.5 или IIS 7 в интегрированном режиме

В ASP.NET 4 относится к изменениям, которые изменяют как **действия** атрибута HTML- **формы** элемент отображается в том случае, когда URL-адрес без расширения преобразует документ по умолчанию. Пример URL-адреса без расширений для разрешения в документ по умолчанию будет [http://contoso.com/](http://contoso.com/), полученный в запросе на [http://contoso.com/Default.aspx](http://contoso.com/Default.aspx).

В ASP.NET 4 теперь отображает HTML- **формы** элемента **действия** значением атрибута пустой строки при выполнении запроса к URL-адрес без расширения, сопоставленного с ним документ по умолчанию. Например, в более ранних версиях ASP.NET, запрос на [http://contoso.com](http://contoso.com) приведет к появлению запрос на `Default.aspx`. В этом документе открывающий **формы** тег будет отображаться как в следующем примере:

`<form action="Default.aspx" />`

В ASP.NET 4 запрос на [http://contoso.com](http://contoso.com) также приводит к запросу к `Default.aspx`. Тем не менее, ASP.NET теперь отображает открывающий HTML **формы** тег, как показано в следующем примере:

`<form action="" />`

Это различие в способе **действия** атрибут отрисовывается может вызвать незначительные изменения в способ обработки службами IIS и ASP.NET формы post. Когда **действия** атрибут является пустой строкой, IIS **DefaultDocumentModule** объект будет создан дочерний запрос `Default.aspx`. В большинстве случаев этот дочерний запрос прозрачен для кода приложения и `Default.aspx` страница выполняется в обычном режиме.

Однако возможное взаимодействие между управляемым кодом и интегрированным режимом IIS 7 или IIS 7.5 может привести к тому, что управляемые ASPX-страницы перестанут работать корректно во время выполнения дочернего запроса. При выполнении следующих условий, дочерний запрос `Default.aspx` документа приведет к ошибке или непредвиденное поведение:

1. ASPX-страницы отправляется в браузер с **формы** элемента **действия** атрибуту присвоено значение «».
2. Форма отправляется обратно ASP.NET.
3. Управляемый HTTP-модуль считывает некоторую часть тела сущности. Например, модуль считывает **Request.Form** или **Request.Params**. Это приводит к тому, что тело сущности запроса POST считывается в управляемую память. В результате тело сущности становится недоступным для модулей машинного кода, запущенных в интегрированном режиме IIS 7 или IIS 7.5.
4. IIS **DefaultDocumentModule** объекта со временем выполняется и создает дочерний запрос `Default.aspx` документа. Однако так как тело сущности уже было считано частью управляемого кода, тело сущности для отправки в дочерний запрос отсутствует.
5. При запуске конвейер HTTP для запроса дочернего элемента, обработчик `.aspx` файлов выполняется на этапе выполнения обработчика.
6. Так как тела функции, переменные не формы и ни одно состояние представления и таким образом сведения недоступны для обработчику ASPX-страницы определить, какие события (если есть) должен вызываться. В результате для соответствующей ASPX-страницы ни один из обработчиков событий обратной передачи не выполняется.

Это поведение можно обойти следующим образом:

- Определите HTTP-модуль, который обращается к тело сущности запроса во время запросов документа по умолчанию и определить, является ли его можно настроить для выполнения только для управляемых запросов. В режиме интеграции с IIS 7 и IIS 7.5 можно пометить модули HTTP, чтобы запустить только для управляемых запросов, добавив следующий атрибут в модуль **system.webServer/modules** входа:

- `precondition="managedHandler"`

- Этот параметр отключает, запрашивает модуль для IIS 7 и IIS 7.5 определить как не управляемый запросов. Для запросов документа по умолчанию первый запрос — URL-адрес без расширения. Таким образом службы IIS не выполняется управляемые модули, которые помечены с предусловия управляемый обработчик во время обработки исходного запроса. В результате управляемые модули не будут случайно прочитать тело сущности, и таким образом тела сообщения по-прежнему доступны и передается вместе в дочерний запрос и документ по умолчанию.

- Если проблемный модули HTTP нужно запускать для всех запросов (для статических файлов без расширений URL-адресов, которые разрешены на **DefaultDocumentModule** объект для управляемого запросы, т. д.), явно изменить затронутых ASPX-страницы, Установка **действия** свойства страницы **System.Web.UI.HtmlControls.HtmlForm** управления непустой строкой. Например, если документ по умолчанию является `Default.aspx`, измените код страницы, чтобы явно задать **HtmlForm** элемента управления **действия** свойства «Default.aspx».

## <a name="changes-to-the-aspnet-code-access-security-cas-implementation"></a>Изменения в реализации разграничения доступа кода ASP.NET

ASP.NET 2.0, и с помощью расширения функций ASP.NET, которые были добавлены в 3.5 использовать .NET Framework 1.1 и 2.0 кода (CAS) модели управления доступом. Однако реализация управления доступом для кода в ASP.NET 4 значительно усовершенствована. В результате приложения ASP.NET с частичным доверием, зависящие от доверенного кода, выполняемого в глобальный кэш сборок (GAC) может завершиться с различные исключения безопасности. Приложения с частичным доверием, опирающиеся на расширенные изменения на компьютере политику разграничения доступа кода также может завершиться с исключения системы безопасности.

Вы можете восстановить ASP.NET 4 приложения с частичным доверием поведение ASP.NET версии 1.1 и 2.0 с помощью нового **legacyCasModel** атрибута в **доверия** элемента конфигурации, как показано в следующем примере :

`<trust level= "Medium" legacyCasModel="true" />`

При восстановлении для устаревшей модели управления доступом для кода, включены следующие варианты поведения старого ЦС:

- Политика разграничения доступа кода Machine учитывается.
- Допускается несколько различных уровней разрешений в одном домене приложения.
- Явное разрешение утверждения не являются обязательными для сборок в глобальном кэше СБОРОК, которые вызываются при только ASP.NET или другой код .NET Framework находится в стеке.

Один из сценариев не может быть восстановлена в .NET Framework 4: веб-приложения с частичным доверием может больше не вызывают некоторых интерфейсов API в System.Web.dll и System.Web.Extensions.dll. В предыдущих версиях платформы .NET Framework можно было для веб-приложения с частичным доверием должны быть явно предоставлены **AspNetHostingPermission** разрешения. Затем можно использовать эти приложения **System.Web.HttpUtility**, типы в **System.Web.ClientServices.\***  пространства имен и типы, связанные с членства, ролей и профилей. Вызов эти типы из приложения с частичным доверием не веб-приложений больше не поддерживается в .NET Framework 4.

> [!NOTE]
> **HtmlEncode** и **HtmlDecode** функциональные возможности **System.Web.HttpUtility** класса был перемещен на .NET Framework 4  **System.Net.WebUtility** класса. Если использовалась только функциональные возможности ASP.NET, который использовался изменить код приложения, чтобы использовать новую **WebUtility** вместо этого класс.


Ниже приведен высокоуровневую сводку изменений для реализации управления доступом для кода по умолчанию в ASP.NET 4.

- Домены приложений ASP.NET теперь являются однородные домены приложений. В домене приложения доступны только наборы разрешений частичного доверия и полным доверием.
- Наборы прав частичного доверия ASP.NET не зависят от любой политики разграничения доступа кода уровня предприятия, компьютера или пользователя.

## <a name="membershipuser-and-other-types-in-the-systemwebsecurity-namespace-have-been-moved"></a>Переместить MembershipUser и других типов в пространстве имен System.Web.Security

Некоторые типы, которые используются в членства ASP.NET были перемещены из `System.Web.dll` на новую сборку System.Web.ApplicationServices.dll. Типы были перемещены для устранения проблемы зависимостей уровней архитектуры между типами в клиенте и в расширенных SKU платформы .NET Framework.

Проектов веб-сайтов проблем не возникает в результате перемещение этих типов, так как System.Web.ApplicationServices.dll был добавлен в список сборок, на которую указывает ссылка, по умолчанию используется системой компиляции ASP.NET. При обновлении проекта веб-сайта, который был создан с помощью более ранней версии ASP.NET в ASP.NET 4, открыв его в Visual Studio 2010 проект компилируется без ошибок.

Аналогичным образом при обновлении проекта веб-приложения, который был создан в более ранней версии ASP.NET в ASP.NET 4, открыв его в Visual Studio 2010, процесс обновления добавляет в проект ссылку на System.Web.ApplicationServices.dll. Таким образом обновляется Web проекты приложений также компилируется без ошибок.

Скомпилированный-файлы, созданные с помощью более ранних версиях ASP.NET также будут работать без ошибок в ASP.NET 4, несмотря на то, что типы членства были перемещены в другой сборке. Тип отправку информации добавлен к версии ASP.NET 4 `System.Web.dll` , автоматически направляет ссылки во время выполнения для этих типов в новое расположение для типов.

Тем не менее библиотеки классов, которые используют типы определенного членства и были обновлены из предыдущих версий ASP.NET не компилироваться при использовании в проекте ASP.NET 4. Например может завершиться неудачно проект библиотеки классов для компиляции и сообщения об ошибке, например следующие:

## <a name="output-caching-changes-to-vary--http-header"></a>Кэширование изменений для изменения вывода \* HTTP-заголовок

В ASP.NET версии 1.0, вызвавшего ошибку кэшированные страницы с `Location="ServerAndClient"` как параметр кэша вывода для порождения `Vary:*` заголовок HTTP в ответе. Таким образом, клиентские браузеры получали указание никогда не кэшировать такую страницу локально.

В ASP.NET версии 1.1 **System.Web.HttpCachePolicy.SetOmitVaryStar** добавлен метод, который можно вызвать для подавления `Vary:*` заголовок. Данный метод был выбран, поскольку изменение порожденную заголовок HTTP считается критическое изменение во время. Однако разработчики были путать с помощью поведения в ASP.NET и предложить отчеты об ошибках, что разработчики не знают о существующих **SetOmitVaryStar** поведение.

В ASP.NET 4 было сделано для устранения неполадки. `Vary:*` Заголовок HTTP больше не создается для ответов, задающих следующую директиву:

`<%@OutputCache Location="ServerAndClient" %>`

В результате **SetOmitVaryStar** больше не требуется для подавления `Vary:*` заголовок.

В приложениях, которые указывают `Location="ServerAndClient"` в **@ OutputCache** директив на странице, теперь вы увидите поведение, содержится в разрешении имя **расположение** значение атрибута — это, будет страниц может быть кэширован в браузере без необходимости вызова **SetOmitVaryStar** метод.

Если страницы в приложении должен предоставлять `Vary:*`, вызовите **AppendHeader** метода, как показано в следующем примере:

`HttpResponse.AppendHeader("Vary","*");`

Кроме того, можно изменить значение кэширования вывода **расположение** атрибут «Server».
