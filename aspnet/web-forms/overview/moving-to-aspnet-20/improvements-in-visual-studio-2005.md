Visual Studio 2005 предоставляет разработчики веб-приложений с большом количестве улучшений и усовершенствований для веб-проектов. Также производительно, как Visual Studio .NET 2002 и 2003, было много жалобы в способе обработки веб-проектов. Visual Studio 2005 добавляет значительное количество новых компонентов для решения этих проблем. Для тех, которые предпочитают способ обработки компиляции веб-приложений в Visual Studio .NET 2003, в разделе [проектов веб-приложений](https://go.microsoft.com/fwlink/?LinkId=57870).

В этот модуль также рассматриваются методы создания веб-проекта, управления и разработки. В более поздней версии модуля также охватывает улучшения в построении веб-проектов и их развертывания.

## <a name="frontpage-server-extensions"></a>Серверные расширения FrontPage

Visual Studio .NET 2002 и 2003 необходимые серверные расширения FrontPage на поле для создания или построения веб-проектов. Разработчики у Выбор между двумя режимами различный уровень доступа (режим доступа серверных расширений FrontPage или файл), оба используются серверные расширения FrontPage для выполнения задач, таких как установка корневой каталог приложения в IIS, и т. д.

Visual Studio 2005 устраняет зависимость от серверных расширений FrontPage для локальных проектов. Visual Studio 2005 теперь осуществляет доступ к метабазе IIS напрямую, а не с помощью серверных расширений FrontPage. Visual Studio 2005 также добавляет поддержку для FTP, позволяющий проект удаленного доступа без использования серверных расширений FrontPage.

Для разработчиков, которым требуется использовать в своих проектах серверные расширения FrontPage параметр по-прежнему доступен. Тем не менее основываясь на надежный отзывах членов сообщества разработчиков ASP.NET, это не является обязательным.

> [!NOTE]
> Серверные расширения FrontPage по-прежнему необходимы для создания удаленного проекта, открытия и т. д.


## <a name="aspnet-development-server"></a>Сервер ASP.NET Development Server

Visual Studio 2005 поставляется с нового веб-сервер, который вызывается ASP.NET Development Server. (Этот веб-сервер был ранее известен как Cassini.)

Существует несколько преимуществ ASP.NET Development Server.

- Теперь возможна, являющиеся администраторами для разработки и отладки для веб-сервера.
- ASP.NET Development Server динамически сопоставляет виртуальных каталогов в любое место в файловой системе, что обеспечивает расположения проектов по гибкой.
- Пользователей в Windows XP Professional, которые уже используются службы IIS теперь можно создать новый веб-приложений, которые не влияют на структуру файла или папки из своего по умолчанию веб-сайта в службах IIS.

Чтобы воспользоваться преимуществами ASP.NET Development Server требуется никакая специальная настройка. Если отлаживается или просматривать веб-проекта, который размещен в файловой системе, Visual Studio 2005 автоматически запускает экземпляр сервера разработки ASP.NET на случайный порт для обслуживания запроса.

Дополнительные сведения будут рассматриваться с ASP.NET Development Server, далее в этом модуле.

## <a name="improved-file-management"></a>Управление файлами

В Visual Studio 2002 и 2003 файл проекта (.vbproj для VB.NET) и .csproj для C# хранятся сведения для всех файлов в веб-приложения. Отображение обозревателя решений основана на сведения о файлах в файле проекта. По этой причине обозреватель решений будет часто отображаются неверные сведения в случаях, где использовались внешних редакторов. Visual Studio 2002 и 2003 будет часто перезаписывать изменения файла или отображает самую последнюю версию файлов.

Visual Studio 2005 немедленно выполняет в этом файле проекта. Вместо этого она считывает данные файлов и папок непосредственно с диска, приведет к точное отображение файлов в проекте. Поскольку папки ссылки в Visual Studio 2002 и 2003 не представляет фактический папки веб-приложения, Visual Studio 2005 также удаляет папку ссылки из обозревателя решений. Для доступа к ссылок проекта в Visual Studio 2005, следует использовать страницы свойств для проекта.

## <a name="lab-setup-ftp-site-and-create-a-project"></a>Лаборатории: Настройка FTP сайта, а также создание проекта

Следующие шаги настройки FTP-сайта, чтобы у пользователя есть только их можно передать по протоколу FTP расположение.

### <a name="install-the-ftp-service"></a>Установить службу FTP

1. Установка и удаление программ, выберите компонентов Windows
2. Выберите Internet Information Services (сервер приложений в Windows 2003) и нажмите кнопку **сведения**.
3. Проверьте **службы протокола передачи файлов (FTP)** и нажмите кнопку **ОК**.
4. Нажмите кнопку **Далее** для установки службы FTP.

### <a name="create-a-new-folder-for-content"></a>Создайте новую папку для содержимого

1. В проводнике Windows создайте новую папку с именем **User1** внутри c:\inetpub\wwwroot.

#### <a name="configure-folders-and-permissions-on-folders"></a>Настройка папки и разрешения для папок.

1. Откройте оснастку служб IIS с помощью средств администрирования. Теперь вы можете папка FTP-узлы в узле имени компьютера.
2. Разверните **FTP-узлов**.
3. Щелкните правой кнопкой мыши **FTP-узел по умолчанию**выберите **New**, затем **виртуальный каталог**, нажмите кнопку **Далее**.
4. Введите **User1** имя виртуального каталога и нажмите кнопку **Далее**.
5. Введите **c:\inetpub\wwwroot\User1** путь и нажмите кнопку **Далее**.
6. Нажмите кнопку **Далее** и затем **Готово** для завершения работы мастера.
7. Щелкните правой кнопкой мыши **User1** виртуальный каталог на FTP-узел по умолчанию и выберите **свойства**.
8. Проверьте **записи** флажок и нажмите кнопку **ОК** чтобы закрыть диалоговое окно.
9. Щелкните правой кнопкой мыши **FTP-узел по умолчанию** и выберите **свойства**.
10. На **учетные записи безопасности** , установите флажок **Разрешить анонимные подключения**.
11. Нажмите кнопку **Да** в диалоговом окне с вопросом, следует продолжить.
12. Нажмите кнопку **ОК** чтобы закрыть диалоговое окно.
13. Разверните **веб-сайт по умолчанию** под **веб-сайтов** узла.
14. Щелкните правой кнопкой мыши **User1** каталог и выберите **свойства**
15. В **параметры приложения** щелкните **создать** пометить папки как приложение.
16. Нажмите кнопку **ОК** чтобы закрыть диалоговое окно.
17. Закройте оснастку служб IIS.

### <a name="create-web-project"></a>Создание веб-проекта

1. Откройте Visual Studio 2005.
2. Из **файл** последовательно выберите пункты **новый веб-сайт**.
3. В **расположение** раскрывающийся список, выберите **FTP**.
4. Нажмите кнопку **Обзор**.
5. Введите **localhost** в **сервера** текстового поля.
6. Введите **User1** в текстовом поле каталога.
7. Нажмите кнопку **откройте**. В диалоговом окне Новый веб-сайт будет введен адрес FTP.
8. Нажмите кнопку **ОК**.
9. Снимите флажок **анонимный вход** в диалоговом окне вход в FTP, введите свои учетные данные и нажмите кнопку **ОК**.
10. Что такое URL-адрес для проекта? (URL-адрес для проекта будет отображаться в обозревателе решений.)
11. Из **построения** последовательно выберите пункты **построить веб-узел** или **построить решение**.
12. Щелкните правой кнопкой мыши на Default.aspx в обозревателе решений и выберите **Просмотр в браузере**.
13. В диалоговом окне требуется URL-адрес веб-сайта, введите `http://localhost/user1` URL-адрес и нажмите кнопку **ОК**.

> [!NOTE]
> Если возникает ошибка, указывающими на невозможность загрузить тип \_по умолчанию, убедитесь, что запущены ASP.NET 2.0 на веб-сайта и не более ранней версии. Это можно сделать на вкладке ASP.NET в службах IIS.

## <a name="improvements-in-compilation"></a>Усовершенствования в компиляции

Работа с веб-приложений в Visual Studio 2005 — заметно быстрее, чем в предыдущих версиях. Причиной этого является в небольшой частью нет изменений в архитектуре компиляции.

В Visual Studio 2002 и 2003 веб-приложений были скомпилированы в один основную сборку, которая находится в папке/Bin. В Visual Studio 2005, приложение\_был добавлен в папку кода. Классы и любой другой код без пользовательского интерфейса добавляются к приложению\_папки с кодом. Если Visual Studio создает проект, все файлы в приложении\_папки с кодом компилируются в одном приложении\_Code.dll файл. Результат этого изменения является намного быстрее, чем в предыдущих версиях последующие сборки.

> [!NOTE]
> Программу командной строки MSBuild может также использоваться для создания приложений ASP.NET Web. Это средство будет рассматриваться в модуль 9.


Другое усовершенствование компиляции является параметром новое построение в меню Построение. Эта функция позволяет разработчику Перестроить только текущей страницы (вместе с, курс и зависимостей), чтобы изменения могут компилироваться быстрее. Поскольку C# не предлагает фоновая компиляция по требованию для целей обновления IntelliSense, т. д., они поможет нам эта функция так, как появится возможность для IntelliSense, чтобы быстро обновить просто перестроения на одной странице.

Свойства построения для проекта позволяют настраивать тип построения, которое происходит перед выполнением стартовой страницы. Разработчики могут выбрать построение только текущую страницу, чтобы быстрее отладку приложений после изменений кода можно запустить Visual Studio.

## <a name="improvements-in-deployment"></a>Усовершенствования в развертывании

Как с помощью Visual Studio 2002 и 2003, Visual Studio 2005 предоставляет возможность Копировать проект. Однако компонент была усилена в Visual Studio 2005 и теперь называется копирования веб-сайта.

Диалоговое окно копирования веб-сайта делится на левой и правой части. Левая часть называется исходный веб-сайт а рамки справа удаленного веб-сайта. Это может быть сайта в локальной файловой системе или на локальном экземпляре служб IIS. Кроме того, сайта, отображаемое в левом окне не обязательно исходного веб-сайта, так как диалоговое окно позволяет публиковать на удаленный веб-сайте *для* исходного веб-сайта.

## <a name="debugging-as-a-non-administrator"></a>Отладка как без прав администратора

Добавление сервера разработки ASP.NET позволяет не только администраторам легко отлаживать приложения ASP.NET сразу после установки. При отладке приложения ASP.NET, работающего в локальной файловой системе Visual Studio запускается сервер разработки ASP.NET в контексте пользователя, выполнившего вход. Этому пользователю можно отлаживать приложения без дополнительной настройки.

## <a name="debug-is-false-by-default"></a>Отладка имеет значение False по умолчанию

В ASP.NET 1.x, *отладки* атрибута в *компиляции* было присвоено значение элемента в файле web.config *true* по умолчанию. Он всегда рекомендации, разработчики присвоить этому атрибуту значение *false* перед развертыванием приложения в рабочей среде, но так как большинство разработчиков не полностью неясны последствия оставить значение атрибута отладки значение true, они просто оставить ее в качестве-является.

Наиболее серьезные проблемы с входящим атрибут debug, равным true — отключает ASP.NETs пакетной компиляции модели. Таким образом каждая страница компилируется в отдельной библиотеке DLL. Если веб-приложение состоит из тысячи страниц (не знал по любым способом), что означает несколько тысяч небольших библиотек DLL создается этим приложением. Хотя эти библиотеки DLL, небольшой размер, они не загружались в любой определенного расположения в памяти. Таким образом они привести к фрагментации в системной памяти и могут участвовать в OutOfMemoryException вхождений.

В ASP.NET 2.0 атрибут debug присвоено значение false по умолчанию. Как вы уже видели, когда разработчик выполняет отладку приложения ASP.NET в Visual Studio 2005, ему будет предложено добавить файл web.config с включенной отладкой. Это приводит к же недостатки, которые присутствовали в ASP.NET 1.x, но теперь разработчик четко предупреждается о том, что атрибут следует восстановить false перед перемещением приложения в рабочей среде.

## <a name="remote-debugging-setup-and-configuration"></a>Установка и настройка удаленной отладки

В Visual Studio 2002 и 2003, удаленная отладка полагается на диспетчер отладки (mdm.exe) и vs7jit.exe процесса. Из-за этого Устранение неполадок удаленной отладки была часто черный прямоугольник для клиентов и часто не было гораздо лучше для службой технической поддержки.
