> Локализация — это процесс проектирования и интеграции поддержки для конкретного языка и региональных параметров в приложение или компонент приложения. Платформе Microsoft ASP.NET предоставляет расширенную поддержку локализации для стандартных приложений ASP.NET благодаря интеграции стандартную модель локализации .NET; Microsoft AJAX Framework использовать интегрированную модель для поддержки различных сценариев, в которых могут выполняться локализации.


## <a name="introduction"></a>Вступление

Технология ASP.NET корпорации Майкрософт предоставляет модель программирования объектно ориентированного и событиями и объединяет его с преимуществами скомпилированного кода. Однако его серверную обработку модели имеет несколько недостатков, присущих технологии, многие из которых можно обратиться по новых возможностей, включенных в пространстве имен System.Web.Extensions, инкапсулирующий службы AJAX корпорации Майкрософт в платформе .NET Framework 3.5. Эти расширения позволяют множество возможностей многофункциональных клиентах, доступны как часть расширения AJAX ASP.NET 2.0, но теперь входит в библиотеке базовых классов Framework ранее. Элементы управления и функции из этого пространства имен включают частичную отрисовку страницы без необходимости обновления всей странице, возможность доступа к веб-служб посредством клиентского скрипта (в том числе ASP.NET, API профилирования) и расширенного клиентского API-интерфейса разработан для отражения, многие из схемы управления, показано в наборе серверный элемент управления ASP.NET.

В этом техническом документе рассматриваются возможности локализации, присутствуют в Microsoft AJAX Framework и скрипт библиотеки Microsoft AJAX, в контексте необходимого для поддержки локализации и просмотра уже интегрирована поддержка локализации в веб- приложения, предоставляемые платформой .NET Framework. Библиотека Microsoft AJAX скрипт использует формату файла RESX уже используемые приложения .NET, который предоставляет встроенную поддержку интегрированной среды разработки и тип совместного использования ресурсов.

В этом техническом документе основано на выпуске бета-версия 2 Microsoft Visual Studio 2008. В этом техническом документе также предполагается, будут работать с Visual Studio 2008, не Visual Web Developer Express и предоставит пошаговые руководства по пользовательского интерфейса Visual Studio. Некоторые примеры кода будет использовать шаблоны проектов, которые могут быть недоступны в Visual Web Developer, экспресс-выпуск.

## <a name="the-need-for-localization"></a>*Потребность в локализации*

Особенно для разработчиков приложений предприятия и разработчики компонентов возможность создания средства, которые могут быть помнить о различиях языков и региональных параметров и языков становится все более необходимые. Разработка компонентов возможность адаптироваться к языкового стандарта клиента повышает производительность разработчика и сокращает объем работ, необходимый для адаптации компонента для функции глобально.

Локализация — это процесс проектирования и интеграции поддержки для конкретного языка и региональных параметров в приложение или компонент приложения. Платформе Microsoft ASP.NET предоставляет расширенную поддержку локализации для стандартных приложений ASP.NET благодаря интеграции стандартную модель локализации .NET; Microsoft AJAX Framework использовать интегрированную модель для поддержки различных сценариев, в которых могут выполняться локализации. С помощью Microsoft AJAX Framework скриптов можно либо локализовать развертываемого во вспомогательные сборки или путем использования структуру системы статических файлов.

## <a name="embedding-scripts-with-satellite-assemblies"></a>*Внедрение скриптов с вспомогательные сборки*

Соответствует стандартной стратегия локализации .NET Framework, ресурсы могут включаться во вспомогательные сборки. Вспомогательные сборки обеспечивает ряд преимуществ над включения традиционных ресурсов в двоичные файлы - любого заданного локализации могут обновляться без обновления более крупного изображения, можно развернуть дополнительный локализации просто путем установки вспомогательных сборок в можно развернуть папку проекта, а вспомогательные сборки не вызывая перезагрузку главной сборки проекта. Особенно в проектах ASP.NET это полезно, так как он может значительно уменьшить объем системных ресурсов, используемых добавочные обновления и как минимум работу производственного использования веб-сайта.

Внедрять скрипты в сборки, включая их в управляемые .resx (или скомпилированные .resources) файлы, которые включены в сборку во время компиляции. Затем становятся доступными для приложения скрипта с помощью AJAX среды выполнения кода, созданного, через атрибуты уровня сборки свои ресурсы

*Соглашения об именовании для внедренных файлов скриптов*

Управление скрипта Microsoft AJAX Framework поддерживает разнообразные варианты для использования в развертывании и тестировании скриптов и рекомендации предоставляются для облегчения этих параметров.

*Для упрощения отладки:*

Сценарии выпуска (производство) не должно содержать `.debug` квалификатор в имени файла. Скриптов, предназначенных для отладки должны содержать `.debug` в имени файла.

*Для упрощения локализации:*

Сценарии для нейтральной культуры, не должно содержать любой идентификатор языка и региональных параметров, имя файла. Для сценариев, которые содержат локализованные ресурсы код языка ISO должны быть указаны в имени файла. Например `es-CO` расшифровывается испанский Колумбии.

В следующей таблице перечислены в файле с примерами контекст именования:

| имя_файла | Значение |
| --- | --- |
| Script.js | Сценарий окончательной версии зависящий от языка и региональных параметров. |
| Script.debug.js | Отладочная версия сценарий, зависящий от языка и региональных параметров. |
| Script.en-US.js | Выпуск версии английский, США сценарий. |
| Script.debug.es-CO.js | Отладочная версия испанский, Колумбия скрипт. |

## <a name="including-a-static-script-file-structure"></a>*Включая структуре файлов скриптов*

При использовании статических файлов скрипта для развертывания, теряются преимущества с помощью встроенного схемы локализации .NET. В основном видимым является потеря автоматический тип, сформированный в том числе файлы скриптов ресурсов; в приведенном выше примере например, ресурсы были доступны с автоматически генерируемым типом, вызвать сообщение из элемента управления ScriptManager.

Однако, существуют некоторые преимущества использования структуре файлов скриптов. Можно обновлять без повторной компиляции и повторного развертывания вспомогательных сборок и использование структуры статических файлов можно также переопределить внедренный скрипт для интеграции дополнительный фрагмент функциональных возможностей, которые могут не поставлялись с компонентом.

Корпорация Майкрософт рекомендует, как избежать проблемы управления версий путем автоматического создания скрипта ресурсов во время компиляции проекта. При обслуживании базовый код расширенный сценарий, может стать труднее отражались изменения кода в каждой локализованной скрипта. В качестве альтернативы можно просто поддерживать один скрипт логики и несколько сценариев локализации объединить файлы при построении проекта.

Так как не ресурсов для включения декларативно, статический сценарий файлы должны быть ссылаетесь, либо путем добавления `<asp:ScriptElement>` элементов как дочерний `<Scripts>` тега элемента управления ScriptManager, либо добавив программно `ScriptReference` объектов Чтобы `Scripts` свойство `ScriptManager` управления на странице во время выполнения.

## <a name="the-scriptmanager-and-its-role-in-localization"></a>*ScriptManager и их роли в локализации*

ScriptManager включает несколько аспектами автоматических действий для локализованных приложений:

- Она автоматически выполняет поиск файлов скриптов на основе параметров и соглашения об именовании; для экземпляра загружает локализации сценариев на основе выбора интерфейс браузера пользователя, и он загружает включена отладка скриптов в режиме отладки.
- Он позволяет определять языков и региональных параметров, включая пользовательские языки и региональные параметры.
- Сжатие файлов сценариев, при этом по протоколу HTTP.
- Он кэширует скрипты для эффективного управления много запросов.
- Он добавляет уровень косвенного обращения скриптов путем передачи их через зашифрованный URL-адрес.

Ссылки на скрипты можно добавить к элементу управления ScriptManager программно или с декларативной разметкой. Декларативная разметка особенно полезен при работе со скриптами, внедренных в сборки, отличную от проекта веб-сайта, как имя сценария Скорее всего не изменит редакции передаются.

## <a name="summary"></a>Сводка

По мере роста веб-приложений для достижения более широкой аудитории необходимость иметь доступ к более широкой языков и региональных параметров и сообществ становится основной бизнес-модели; электронной коммерции веб-приложений необходимо уметь работать с иностранных валют, должны существовать возможность не только их содержимое, но также их указания навигации и полей формы в другие языки и компаний необходимо знать, что данная операция системы управления содержимым возможен доступ.

Платформа .NET Framework поддерживает по своей природе является структурой широкие возможности локализации, вспомогательные сборки и XML-файлов ресурсов (RESX) для представления единообразный способ поиска строк ресурсов и изображений. Расширения ASP.NET AJAX, включая Microsoft AJAX Framework и скрипт библиотеки Microsoft AJAX, обеспечивают поддержку этой модели программирования в код на стороне клиента, включение простой поиск строки. Вспомогательные сборки поддерживает автоматическое включение скрипт ресурсов (фактические JS-файлов) через ScriptResource.axd при условии, что имена файлов следуйте данной схемы именования. Эта поддержка расширения ASP.NET AJAX упростить скриптов локализации и глобализации приложений.
