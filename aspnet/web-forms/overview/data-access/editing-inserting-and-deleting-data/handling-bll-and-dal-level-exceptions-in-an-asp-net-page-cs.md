## <a name="introduction"></a>Вступление

Работа с данными из веб-приложения ASP.NET с помощью архитектуры многоуровневое приложение включает в себя следующие три действия:

1. Определите, какой метод уровня бизнес-логики необходимо вызвать и какие значения параметров для ее передачи. Значения параметров могут быть жестко заданный, назначенный программным способом или входные данные, введенные пользователем.
2. Вызовите метод.
3. Обработка результатов. При вызове метода уровень бизнес-ЛОГИКИ, который возвращает данные для этого может потребоваться привязка данных к данным веб-элемента управления. Для методов уровень бизнес-ЛОГИКИ, изменяющие данные это может включать выполнение некоторых операций на основе возвращаемого значения или корректно обработке любых исключений, возникших на шаге 2.

Как мы видели в [с предыдущим учебником](examining-the-events-associated-with-inserting-updating-and-deleting-cs.md), ObjectDataSource и данных веб-элементы управления предоставляют точки расширяемости для шагов 1 и 3. GridView, например, запускает его `RowUpdating` событий до присвоения значения его полей методы `UpdateParameters` коллекции; его `RowUpdated` событие возникает после завершения операции ObjectDataSource.

Мы уже рассмотрели события, которые запускаются во время шага 1 и было показано, как их можно использовать для настройки параметров ввода или отменить операцию. В этом учебнике мы будем можем обратиться к события, которые срабатывают после завершения операции. Эти обработчики мы можем, помимо прочего определить, если возникло исключение во время операции и верно его обработать, отображение понятное, информативное сообщение на экране, а не по умолчанию используется стандартная ASP.NET страница «исключение».

Чтобы продемонстрировать работу с этими событиями уровня после, давайте создадим страница, на которой перечислены продукты в изменяемого элемента управления GridView. При обновлении продукта, если исключение вызвано ASP.NET страница отобразит короткое сообщение над элементом управления GridView, о том, что возникла проблема. Давайте начнем!

## <a name="step-2-gracefully-handling-dal-level-exceptions"></a>Шаг 2: Корректно обработка исключений уровня DAL

Хотя наши изменяемого элемента управления GridView прекрасно работает, когда пользователи вводят допустимые значения для имени измененного продукта, цену и единиц на складе, введя недопустимые значения приводит к возникновению исключения. Например, пропуск `ProductName` приводит [NoNullAllowedException](https://msdn.microsoft.com/library/default.asp?url=/library/cpref/html/frlrfsystemdatanonullallowedexceptionclasstopic.asp) поскольку у `ProductName` свойство в `ProdcutsRow` класс имеет его `AllowDBNull` свойство `false`; Если База данных не работает, `SqlException` будут созданы методом адаптера таблицы, при попытке подключения к базе данных. Без выполнения действий, эти исключения поднимаются с уровня доступа к данным на уровне бизнес-логики, а затем на страницу ASP.NET и, наконец, чтобы среда выполнения ASP.NET.

В зависимости от того, как настроить веб-приложение и посещается ли приложение от `localhost`, необработанное исключение может привести к странице универсального ошибка сервера, отчет подробные сведения об ошибке или удобной веб-страницы. В разделе [приложения Web Error Handling in ASP.NET](http://www.15seconds.com/issue/030102.htm) и [customErrors элемент](https://msdn.microsoft.com/library/h0hfz6fc(VS.80).aspx) Дополнительные сведения о том, как среда выполнения ASP.NET отвечает на неперехваченное исключение.

Рис. 6 показан экран, при попытке обновления продукта без указания `ProductName` значение. Это значение по умолчанию подробный отчет об ошибках, отображаемый при проходе через `localhost`.


[![Пропуск сведения об исключении будут отображаемое имя продукта](handling-bll-and-dal-level-exceptions-in-an-asp-net-page-cs/_static/image17.png)](handling-bll-and-dal-level-exceptions-in-an-asp-net-page-cs/_static/image16.png)

**Рис. 6**: пропуск продукта имя будет отображения сведения об исключении ([Просмотр полноразмерное изображение](handling-bll-and-dal-level-exceptions-in-an-asp-net-page-cs/_static/image18.png))


Хотя такие подробности полезны при тестировании приложения, представивший конечному пользователю экран при возникновении исключения не является идеальным. Скорее всего, конечный пользователь не знает, что `NoNullAllowedException` является или почему оно было вызвано. Оптимальный подход — предоставить пользователю более наглядное сообщение о том, что возникла проблема при попытке обновления продукта.

Если исключение возникает при выполнении операции, события уровня после ObjectDataSource и веб-управления данными предоставляют возможность обнаружит его и отменяет исключение всплывающих окон среды выполнения ASP.NET. Например, давайте создадим обработчик событий для элемента GridView `RowUpdated` событие, определяет ли исключение возникло и если да, отображает сведения об исключении в веб-управления Label.

Начните с добавления метки к странице ASP.NET, установка его `ID` свойства `ExceptionDetails` и очищая его `Text` свойство. Чтобы нарисовать глаз пользователя на данное сообщение, задать его `CssClass` свойства `Warning`, — это класс CSS, мы добавили в `Styles.css` файл в предыдущем учебнике. Напомним, что этот класс CSS вызывает отображенный на красный, курсив, полужирный и долгосрочного текста метки.

## <a name="summary"></a>Сводка

Во время вставки, обновления и удаления операций веб-управления данными и участвующих ObjectDataSource запускаются событий предварительного и последующего уровней, служащих обложкой текущей операции. Как было показано в этом учебнике и предыдущим, при работе с изменяемого элемента управления GridView GridView `RowUpdating` вызывается событие, следуют ObjectDataSource `Updating` событие, после чего команды update выполняется в элемент управления ObjectDataSource базовый объект. После завершения операции, ObjectDataSource `Updated` вызывается событие, а затем в GridView `RowUpdated` событий.

Можно создать обработчики событий для события предварительно уровня, чтобы настроить входные параметры или уровня после события проверки и реагировать на результаты операции. Обработчики наиболее часто используются для определения, произошла ли исключение во время операции. При возникновении исключения, эти обработчики могут при необходимости обработать по отдельности. В этом учебнике мы узнали, как обработка такого исключения, отображая понятное сообщение об ошибке.

В следующем уроке мы рассмотрим, как уменьшить вероятность возникновения исключений, вызываемых вопросы форматирования данных (такие как ввод отрицательной `UnitPrice`). В частности мы рассмотрим способы добавления элементов управления проверки к интерфейсам редактирования и вставки.
